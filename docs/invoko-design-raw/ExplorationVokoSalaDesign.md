# Desktop Exploration 设计稿：Voko / Sala

> 2026-04-09 更新：文中关于 `suggestion` 行和 dashboard suggestion 的设计描述已不再代表当前实现方向。探索页现阶段保留任务/画像主结构，不再展示 suggestion 入口。

## 目标
- `Exploration` 不应该只是一个空白 detail 区，而是 Invoko 应用内两种完全不同的“人物 / 视角”。
- `Voko` 负责当下的交互、回忆、检索、倾听，像应用内一个可以直接对话的人。
- `Sala` 负责总结、映射、反思，把一天的历史记录和 memory 做成“日度画像”。
- 第一阶段先做高保真 mock，建立节奏和人格感，再接真实数据。

## 设计语言总览

### 整体审美方向
- `Exploration` 不走“通用 SaaS dashboard”路线，也不走“霓虹 AI 控制台”路线。
- 视觉方向应该是：
  `一个应用内世界观中的两个房间`
- `Voko` 是一个有舞台感、有人物感、会回应你的房间。
- `Sala` 是一个更冷静、更像工作年鉴和个人日报的房间。

### 共享母体风格
- 基底不是纯白，而是轻微偏暖的雾面中性色。
- 不大面积用玻璃，不做发光描边，不做赛博紫蓝。
- 阴影要短、软、低对比，像纸张和物件浮在桌面上，而不是悬浮卡片。
- 所有模块都要有“留白和静气”，不要把 detail 塞满。

### 共享颜色基调
实现时可以映射成动态颜色，但视觉目标建议如下：

| 语义 | 建议色 |
|---|---|
| 背景底色 | `#F4F1EA` |
| 一级文字 | `#201C18` |
| 二级文字 | `#6A625A` |
| 分隔线 | `#D8D0C7` |
| 高亮暖色 | `#D47A5A` |
| 高亮雾蓝 | `#7C93A8` |
| 高亮橄榄 | `#7F8A63` |

### 共享排版语言
- 优先沿用 macOS 的 `SF Pro` 体系，但不要全靠默认字重堆层级。
- 标题更像“编辑标题”，不是列表标题。
- 正文和说明要更克制，减少过长段落。

建议字号层级：

| 层级 | 建议 |
|---|---|
| 页面标题 | `28-34 / semibold` |
| 模块标题 | `17-20 / semibold` |
| 强调数字 | `30-44 / medium or semibold` |
| 正文 | `13-15 / regular` |
| 辅助注释 | `11-12 / medium` |

### 共享空间语言
- Detail 区不应该是满宽铺开，而应该有明显“舞台边界”。
- 推荐内容最大阅读宽度控制在 `960-1160` 之间。
- 重要元素之间拉开大间距，小元素内部再密一点，形成呼吸节奏。
- 不要让每个模块都长得像同一张 card。

建议间距 token：

| token | 值 |
|---|---|
| `space-1` | `6` |
| `space-2` | `10` |
| `space-3` | `14` |
| `space-4` | `20` |
| `space-5` | `28` |
| `space-6` | `40` |
| `space-7` | `56` |

### 共享圆角与边框
- 小 chip: `12`
- 普通 bubble / 小模块: `18-20`
- 大面板: `26-32`
- 圆形角色核心: `999`
- 边框统一细线，不超过 `1px`

### 共享动效语言
- 动画应该像呼吸、漂浮、收音、排版切换，而不是炫目的 UI 特效。
- 时长以 `180ms / 240ms / 420ms` 为主。
- easing 优先 `easeOut` 和 `spring(response: 0.35-0.5, dampingFraction: 0.8-0.9)`。
- 任何持续动画都必须轻，不抢内容。

### 明确避免
- 大面积毛玻璃卡片堆满屏幕
- 紫蓝霓虹渐变
- 到处发光的圆环和波纹
- 每个模块都是同一种圆角卡片
- 看起来像 AI 生成的通用“智能助手面板”

## 总体原则
- 两个页面共用同一个 sidebar shell，但 detail 区必须一眼看出不是同一套模板。
- `Voko` 是“角色中心”的界面，重心在人物、气泡、对话、围绕感。
- `Sala` 是“总结中心”的界面，重心在结构、比例、趋势、洞察。
- 两者都要优先复用 Invoko 已有语义：
  `History`、`Collection / Memory`、`Running Task`、`Learned Memory`。

## 布局基准

### 以 detail 可用宽度为准
不要以整个窗口宽度为准，因为 sidebar 已经吃掉一部分空间。
下面所有布局都以 `detail` 区域实际可用宽度为准。

| 档位 | detail 宽度 | 用法 |
|---|---|---|
| `Wide` | `>= 1100` | 完整布局，左右和上下关系都成立 |
| `Standard` | `860 - 1099` | 主布局不变，但模块稍微收紧 |
| `Compact` | `640 - 859` | 减少横向并列，改为分层堆叠 |

### 通用外边距
- 顶部安全区以下首屏内容起点：`32-40`
- 左右内容边距：`32`，紧时可降到 `24`
- 模块之间的主间距：`24 / 28 / 40`
- 页内所有内容应在视觉上居中，不贴边铺满

### 通用内容框
- 建议 detail 内再放一个居中的内容框
- `Wide` 最大内容宽度：`1160`
- `Standard` 最大内容宽度：`980`
- `Compact` 最大内容宽度：`760`

### 通用首屏结构
- 顶部不需要传统 toolbar 式信息条
- 页面标题可以非常轻，甚至只占一行
- 首屏的重点应直接进入主视觉，不要标题下面先塞说明段落

## Voko

### 产品定位
`Voko` 不是一个普通聊天页，而是 Invoko 里的“当班角色”。
他应该给人一种感觉：我点开后，不是在看列表，而是在面对一个知道我最近做过什么、也能替我翻历史的人。

### 关键词
- 人物感
- 近身陪伴
- 语音优先
- 记忆回溯
- 任务围绕

### UI 设计语言
- 视觉方向：
  `桌面上的小舞台`
- 不是聊天软件，不是 dashboard，而像“一个角色站在记忆和任务中央”。
- 画面中心一定要强，四周的信息像被这个角色吸附住。

### Voko 的色彩

| 语义 | 建议色 |
|---|---|
| 中央舞台光晕 | `#F7E7D7` |
| 角色核心圆 | `#F3EEE7` |
| 核心描边 | `#D9CDBE` |
| 主强调 | `#C96C4A` |
| 记忆气泡 | `#E6EFE7` |
| 历史气泡 | `#EAE4F3` |
| 任务气泡 | `#F4E2D5` |

### Voko 的构图语言
- 中央角色核心占据 detail 视觉中心的 `35%-40%` 注意力。
- 背后有一个很淡的椭圆形舞台光区，而不是纯平背景。
- bubble 不是网格排布，而是“近轨道环绕”。
- 底部 transcript strip 像舞台前沿的一条字幕带。

### Voko 的形态语言
- 中央圆要大，建议直径 `172-208`。
- 中央圆不能只有一个死板边界，应该至少有：
  `实体圆 + 很轻的外晕 + 一层状态环`
- task/history/memory bubble 不做标准胶囊统一规格。
- bubble 应该有轻微尺寸差：
  `S 96-120`
  `M 132-164`
  `L 180-220`
- bubble 角半径尽量大，但不是完全胶囊，建议 `18-24`。

### Voko 的材质语言
- 中央圆是“陶瓷 / 石膏”感，柔和、哑光。
- bubble 更像压在舞台上的纸片和提示卡，不是玻璃。
- 可使用非常轻的内阴影或顶光，让中央圆更像实体。

### Voko 的排版语言
- 中央角色下方的一句提示语要像角色对白，不像功能说明。
- quick ask chip 文案短、直接、像你会真的对他说的话。
- transcript 文字要比普通说明更大一点，像现场字幕。

建议：
- 中央提示语：`15-17 / medium`
- transcript：`16-18 / regular`
- bubble 标题：`12-13 / semibold`
- bubble meta：`11 / medium`

### Voko 的动效语言
- Idle:
  中央圆轻微呼吸，幅度不超过 `1.5%`。
- Bubble:
  每个 bubble 允许非常轻的错位漂移，周期在 `5-8s`。
- Listening:
  中央环不是波形 UI，而是像“耳朵张开”的双层扩张。
- Enter listening:
  bubble 要稍微往外退 `8-14pt`，给角色让出空间。
- Responding:
  一两个相关 bubble 被点亮，不要全屏一起动。

### Voko 的声音状态样式
- `Idle`
  中央圆平静，底部显示 quick asks。
- `Listening`
  中央圆外圈加厚，出现柔和暖色脉冲。
  transcript strip 从底部滑入。
- `Silence countdown`
  中央外环慢慢收紧，提示“快结束了”。
- `Responding`
  底部 transcript strip 转为 reply strip，颜色从暖白切到微浅棕。

### Voko 的组件口径

| 组件 | 建议 |
|---|---|
| 中央圆 | `172-208` 直径 |
| 外环厚度 | `6-10` |
| bubble 宽 | `96-220` |
| bubble 高 | `40-72` |
| quick ask chip 高 | `32-36` |
| transcript strip 高 | `64-88` |

### Voko 的情绪参照
- 像一个温和的馆员
- 像一个认识你工作习惯的搭档
- 像一个在桌面中央抬头听你说话的人
- 不是科幻 AI 球，不是 Siri，不是客服机器人

### Voko 布局设计
核心目标只有一个：
让用户一打开就先看到“角色”，再看到“围绕他的上下文”，最后才看到“可操作的文字”。

### Voko / Wide 布局

```text
┌────────────────────────────────────────────────────────────────────────────┐
│ page title: Voko                                                          │
│                                                                            │
│         history bubble                 running bubble                      │
│                                                                            │
│   memory bubble            ╭────────────────────╮         history bubble   │
│                            │                    │                          │
│                            │         ○          │                          │
│                            │                    │                          │
│                            │ "Ask me anything"  │                          │
│                            ╰────────────────────╯                          │
│                                                                            │
│         memory bubble                  running/history bubble              │
│                                                                            │
│        [总结我今天干了啥] [找刚刚那条记录] [我最近记住了什么]              │
│                                                                            │
│  transcript / reply strip                                                  │
└────────────────────────────────────────────────────────────────────────────┘
```

### Voko / Wide 的具体区块
1. 顶部标题区
   只保留一个轻标题 `Voko`，左对齐。
   下面不要接副标题。
   标题区总高度控制在 `36-52`。
2. 中央舞台区
   占首屏高度的核心位置，建议高度 `440-560`。
   中央圆位于整个内容框的几何中心略偏上，大约在 `Y = 42%`。
3. 气泡环绕区
   bubble 不是独立模块区，而是属于中央舞台的一部分。
   最大轨道半径大约是中央圆外沿向外 `220-300`。
4. Quick ask 区
   放在中央舞台下方，居中排一行。
   不要贴着 transcript strip。
5. Transcript strip
   作为整页底部的最后一级，横向贯穿内容框。
   不要做成贴边小浮窗。

### Voko / Wide 的定位规则
- 中央圆始终正中偏上，不跟随 bubble 数量乱跑。
- bubble 最多 `6` 个，建议固定六个方位，不要实时重新洗牌：
  `10点 / 1点 / 8点 / 4点 / 11点近侧 / 2点近侧`
- `running task` 永远优先放在右上近侧，离角色最近。
- `最近 history` 优先放左上。
- `memory` 优先放左下或右下。
- bubble 之间不能相互重叠，也不能压住 quick ask 区。

### Voko / Wide 的层级顺序
- 第一视觉层：中央圆
- 第二视觉层：最近且重要的两个 bubble
- 第三视觉层：其他 bubble
- 第四视觉层：quick ask chips
- 第五视觉层：transcript / reply strip

### Voko / Wide 的尺寸关系
- 中央圆：`192`
- 状态外环最大到 `228`
- 舞台光晕椭圆：宽约 `420-520`，高约 `280-340`
- bubble 小号：`100 x 44`
- bubble 中号：`144 x 56`
- bubble 大号：`196 x 68`
- transcript strip：宽度撑满内容框，高度 `72-88`

│              history bubble   running bubble         │
│                                                      │
│                      ○                               │
│               "Ask me anything"                      │
│                                                      │
│           memory bubble      memory/history          │
│                                                      │
│ [总结我今天干了啥] [找刚刚那条记录]                   │

│                                                      │
│ transcript / reply strip                             │
└──────────────────────────────────────────────────────┘

### Voko / Compact 的调整原则
- 中央舞台还在，但 bubble 数量减到 `4`。
- bubble 不再做完整轨道，更像“围在周围的四张纸片”。
- quick ask 改成两行排列。
- transcript strip 仍然全宽，但高度可以降到 `64-76`。
- 不能在窄宽度下把 Voko 退化成普通聊天页。

### Voko 的点击后布局变化
- 点击中央圆进入 listening 时，中央圆不移动。
- bubble 整体外退一圈，留出中央收音感。
- quick ask 区整体淡出，不要保留。
- transcript strip 从底部上浮到可见位置。
- listening 结束后，quick ask 不立刻回来，要先让 reply strip 占住空间。

### Voko 的交互区边界
- 中央圆是最大点击热点。
- bubble 也可点击，但 hover 反馈弱于中央圆。
- quick ask chip 的点击反馈应该更轻，不要像主 CTA。
- transcript strip 不应该吞掉中央舞台的视觉注意力。

### 页面结构

```text
┌──────────────────────────────────────────────────────────────┐
│ Voko                                                        │
│                                                              │
│                    ( orbiting task bubbles )                 │
│             ○                                                │
│        (history)   (running)   (memory)                      │
│                                                              │
│              “Ask me what you did today.”                    │
│                                                              │
│   [总结我今天干了啥] [找刚刚那条记录] [我最近记住了什么]         │
│                                                              │
│   transcript / reply strip                                   │
└──────────────────────────────────────────────────────────────┘
```

### 核心布局
1. 中央舞台
   中间是一个人物占位，第一版先用一个大圆代替人物形象。
   这个圆不应该像按钮，而更像角色头像 / 头部轮廓。
2. 轨道层
   正在运行的 task、重要 history、最近 memory 以气泡形式围绕中央圆分布。
   不是规则仪表盘，而是有一点“漂浮在人物身边”的感觉。
3. 底部动作层
   放 3 到 4 个快速意图 chip，帮助用户理解这个角色能做什么。
4. 最底部转录 / 回应层
   语音聆听时显示实时转录。
   结束后显示一条被确认的用户说话内容，以及 Voko 的简短回应或建议。

### 气泡定义
- `Running task bubble`
  展示当前任务名、状态、时长。
  视觉上最活跃，轻微呼吸感。
- `History bubble`
  展示最近线程或最近一次高价值操作。
  点击后未来可跳转相关 history。
- `Memory bubble`
  展示被保存的 memo / learned memory / long recording summary 的摘要。
  文案更短，更像“你之前记住了这个”。

### 气泡排布原则
- 不做机械圆环平均分布。
- 维持 5 到 7 个 bubble 的“近轨道”效果，大小有差异。
- 距离中央圆最近的是 `running task`。
- 较远轨道放 `history` 和 `memory`。
- bubble 允许轻微漂移，但不应持续高频运动。

### 中央圆状态

| 状态 | 中央圆表现 | 周围气泡 | 底部区域 |
|---|---|---|---|
| Idle | 纯净圆形，轻微呼吸 | 缓慢漂浮 | 展示 quick asks |
| Hover | 外圈提亮，像角色注意到你 | 轻微减速 | 出现 “Tap to talk” |
| Listening | 外环展开，像在收音 | 气泡轻微后退，给中央让位 | 实时转录出现 |
| Processing | 外环收紧，内部微弱脉冲 | 气泡稳定 | 转录锁定，显示等待 |
| Responding | 中央圆恢复稳定 | 一到两个 bubble 被高亮 | 显示总结 / 建议 |

### 点击中央圆后的 mock 交互
这是第一阶段必须实现的“假语音”节奏，先不接真实语音引擎。

1. 点击中央圆，进入 `Listening`。
2. 立即出现转录条，文案从空变成 mock 实时识别。
3. 前 `3s` 连续出字，模拟用户在说话。
4. 接下来 `2s` 不再新增转录，进入静默等待。
5. 满 `2s` 后自动结束聆听。
6. 结束后把本次转录固化成一条用户输入，并给出一条 Voko 的回应或建议。

### mock 时间线

| 时间 | 行为 |
|---|---|
| `t=0s` | 点击中央圆，进入 listening，气泡外撤一点 |
| `t=0.6s` | 出现第一段转录，例如“帮我” |
| `t=1.5s` | 补第二段，例如“回顾一下” |
| `t=2.7s` | 补第三段，例如“我今天都做了什么” |
| `t=3.0s` | 停止新增文字，进入 2 秒 silence countdown |
| `t=5.0s` | 自动结束 listening，转成一条完整 query |

### Voko 的回应方向
第一版不需要长回答，但要显出“他懂应用内历史和 memory”。

建议优先支持的回应类型：
- “你今天主要做了 3 件事”
- “最近的历史记录在这里”
- “你今天在 Xcode / Chrome / Terminal 上切换很多”
- “我记得你保存过这条 memo”
- “要不要我把今天的工作压成一个 summary”

### 快速意图 chip
- `总结我今天干了啥`
- `找刚刚那条历史记录`
- `我最近记住了什么`
- `下午在做哪个任务`

### 视觉语气
- 角色应该是温和、可信、稍微有点拟人，而不是炫技机器人。
- 中央圆建议偏暖白或浅灰，不要荧光，不要科幻蓝紫。
- bubble 应该像“贴身信息”而不是 dashboard 小卡片。
- 整个页面可以有轻微舞台感，中央更亮，四周更安静。

### 数据映射建议
第二阶段接真实数据时，优先使用：
- `RunningTaskStore` 提供当前 task
- `HistoryStore` 提供最近线程 / 今天干了什么
- `CollectionStore` 提供 memo / long recording / fortune / memory 展示
- `UserLearnedMemoryWriter` 产出的 learned memory 作为“我记住了什么”

### 第一阶段不做
- 真实人物插画或 3D 角色
- 真实语音识别
- 复杂任务拖拽
- bubble 深层跳转联动

## Sala

### 产品定位
`Sala` 更像一个“总结性的 memory 空间”，不是角色，而是一种回顾视角。
它应该接近“手机屏幕时间 + 个人工作画像 + suggestions”的混合体。

### 关键词
- 总结
- 反思
- 比例感
- 日度画像
- 洞察

### UI 设计语言
- 视觉方向：
  `工作年鉴 / 日报摘要 / 个人屏幕时间册页`
- 不是角色，也不是操控台。
- 要有编辑性、统计感、排版感，像你在翻一页“今天的自己”。

### Sala 的色彩

| 语义 | 建议色 |
|---|---|
| 纸面背景 | `#F6F2EB` |
| 模块底色 | `#F0EBE2` |
| 墨色文字 | `#1F1B17` |
| 次要说明 | `#72685F` |
| 图表蓝灰 | `#7A8EA1` |
| 图表鼠尾草绿 | `#899377` |
| 图表陶土色 | `#B97A5B` |
| 结构线 | `#D9D1C8` |

### Sala 的构图语言
- 建议采用不完全对称的编辑布局，而不是均匀卡片宫格。
- 顶部 hero 和右侧 app 占比要形成主次对照。
- 中段 suggestion 和 hexagon 形成“结论 + 解释”的组合。
- 底部 highlights 像附录或摘录，不要再做成卡片森林。

### Sala 的形态语言
- 模块更像排版区块，不像独立 app widget。
- 允许使用细分割线、竖线、顶部小标题、注释编号。
- 图表区域边界应克制，尽量靠留白和轻分隔来组织。
- 六角图可以置于一个偏大的留白区域中，给它仪式感。

### Sala 的材质语言
- 更接近纸张、印刷、年鉴，而不是玻璃和塑料。
- 局部可以有非常轻的纸感底纹，但不要真的做明显纹理噪点。
- 图表使用实色块和细线，不使用高饱和渐变。

### Sala 的排版语言
- 标题应该像编辑标题，有明显字号和字重对比。
- 数字和占比可以更大，但不做夸张 KPI 风。
- 小说明文字可以贴边排，形成“批注感”。

建议：
- 顶部一句话 hero：`30-36 / semibold`
- 比例数字：`26-40 / medium`
- 模块标题：`16-18 / semibold`
- 注释：`11-12 / medium`

### Sala 的图表语言
- `App 占比`
  更像 Screen Time，而不是商业 BI 图。
  可用环图或横向堆叠条，但标签必须可读。
- `Ability Hexagon`
  线条要细，填充透明度低，重点是轮廓和差异。
- `Mode tags`
  像编辑标签，不像 filter pill。

### Sala 的动效语言
- 模块入场以渐显和轻位移为主。
- 图表绘制动画应短，像在“描出来”，不是弹出来。
- hover 反馈弱于 Voko。
- Sala 应该更稳，不应该有持续漂浮物。

### Sala 的组件口径

| 组件 | 建议 |
|---|---|
| 顶部 hero 区高度 | `180-240` |
| app 占比面板 | `280-360` 宽 |
| suggestion 块 | `140-200` 高 |
| hexagon 区块 | `320-420` 高 |
| memory highlight 行高 | `68-96` |

### Sala 的情绪参照
- 像一页个人工作日报
- 像手机屏幕时间，但更有性格
- 像一本年鉴里的“今日剖面”
- 不是仪表盘，不是任务列表，不是第二个 collection 页面

### Sala 布局设计
核心目标：
让用户像“翻开一页总结册”一样，自上而下完成今天的理解。
不是到处点击探索，而是被页面带着读完一页。

### Sala / Wide 的网格建议
- 用 `12` 列逻辑网格，但不要画出明显栅格感。
- 推荐列分配：
  hero `1-7`
  app share `8-12`
  suggestion 三块平均分布整行
  hexagon `1-7`
  personality `8-12`
  highlights `1-12`

### Sala / Wide 的阅读顺序
1. Hero 先给结论
2. App share 证明这件事
3. Suggestion 告诉用户怎么理解
4. Hexagon 和 mode tags 把“今天的你”人格化
5. Highlights 给出今天留下来的东西

### Sala / Wide 的模块关系
- Hero 不是 banner，而是整页的总论。
- App share 是英雄模块的证据区。
- Suggestion 必须在视觉上比 hero 小一级，像编辑批注。
- Hexagon 不是 KPI 图，而是“人格轮廓”。
- Highlights 是页尾归档感区域，要更平、更安静。

### Sala / Wide 的尺寸关系
- Hero 高度：`200-240`
- App share 模块高度：`220-280`
- Suggestion 单块高度：`132-168`
- Hexagon 模块高度：`340-420`
- Personality 模块高度：与 hexagon 对齐
- Highlights 区最少显示 `3` 条，整体高度 `260-360`

### Sala / Wide 的区块形状
- Hero：大块留白区，边界感最弱
- App share：半结构化图表模块，可有轻边界
- Suggestion：三块短评模块，像侧注卡片
- Hexagon：单个大区块，边界清晰但不厚重
- Highlights：列表型长区块，行与行之间用细分隔

### Sala / Compact 的调整原则
- 所有主模块改为纵向堆叠。
- Suggestion 保留双列，如果再窄就改单列。
- Hexagon 仍然保留大留白区，不能缩成小图标。
- Highlights 改为单列时间线式摘要。
- 即使压缩，也要保持“总结册页”的阅读顺序。

### Sala 的模块内部布局

#### Hero
- 左上是小标签：
  `Today’s Mode`
- 主句占据绝对视觉重心。
- 主句下面是两到三行总结。
- 右下角可放一个小号统计摘要，不要抢主句。

#### App Share
- 图表在上，标签在下。
- 上半区：环图或横向堆叠条
- 下半区：前四个 app 的名称、占比、时长
- 右侧或底部补一句判断，例如：
  `今天主要时间花在 build 和浏览上`

#### Suggestion
- 每块 suggestion 只说一件事。
- 上面是一个短标题，下面是一句解释。
- 文字不宜超过三行。

#### Ability Hexagon
- 六边形本体放中间偏左或正中。
- 周围不要堆太多标签。
- 只保留六个维度名，外加一个很短解释句。

#### Personality / Mode Tags
- 顶部是 `Today you felt like`
- 下方是 2 到 4 个成对标签
- 下面可以补一段一句话解释

#### Highlights
- 每一行由三部分组成：
  时间 / 类别
  核心摘要
  右侧轻量 meta
- 不要每条都用大卡片包起来

### Sala 的布局红线
- 不要把 hero、chart、hexagon 都做成一样大的卡片。
- 不要让 suggestion 和 highlights 混成一个内容列表。
- 不要让页面缺少“自上而下读完”的叙事性。
- 不要把 Sala 做成第二个 dashboard 首页。

### 页面结构

```text
┌──────────────────────────────────────────────────────────────┐
│ Sala                                                         │
│                                                              │
│  今日画像 hero        应用占比 / 时间分布                     │
│                                                              │
│  suggestion 1         suggestion 2                           │
│                                                              │
│  ability hexagon      personality / MBTI-like traits         │
│                                                              │
│  key memories / highlights timeline                          │
└──────────────────────────────────────────────────────────────┘
```

### 核心模块
1. 日度 Hero
   顶部先给一个一句话判断，例如：
   `今天是偏 Builder 的一天。`
   或：
   `今天大部分时间花在产出，而不是切换。`
2. 应用占比
   参考手机 `Screen Time` 的感觉，展示今天主要时间消耗在哪些 app。
   可做成环形占比或竖向堆叠条，但气质要平静、克制。
3. Suggestion 区
   不是命令式 CTA，而是像“你可以这样理解今天”。
4. Ability Hexagon
   用六边形 / 雷达图呈现当天的工作特征，而不是单纯效率分数。
5. Memory Highlights
   把今天最值得记住的 3 到 5 条内容做成摘要流。

### App 占比建议
第一版可以先 mock：
- Xcode
- Chrome
- Terminal
- Figma / Cursor / Notion 之类的其他应用

展示口径建议：
- 总时长占比
- 切换频率
- 深度工作连续时段

### Suggestion 文案方向
- `你今天 42% 的时间在构建，只有 11% 花在回顾。`
- `下午两点后上下文切换明显增多。`
- `你保存了 3 条 memory，但还没有把它们收束成总结。`
- `今天的工作更像“推进”而不是“探索”。`

### Ability Hexagon 维度建议
六个维度建议固定，不要每天乱变：
- Focus
- Output
- Exploration
- Recall
- Follow-through
- Recovery

解释：
- `Focus` 看连续工作时段
- `Output` 看完成和落地产出
- `Exploration` 看搜索、切换、尝试
- `Recall` 看 history / memory 被调用与沉淀程度
- `Follow-through` 看任务闭环程度
- `Recovery` 看节奏是否过于拥挤，是否有停顿

### MBTI-like 呈现建议
不要真的做人格测试，但可以借用“人格标签”的呈现方式。

建议形式：
- 一个 `Today’s Mode` 标签
- 配两到四个相对维度

例如：
- `Builder / Explorer`
- `Deep Work / Reactive`
- `Solo / Collaborative`
- `Closer / Wanderer`

这部分更像“今天你的工作人格偏向哪里”，而不是永久人格。

### Memory Highlights 区
这里是 Sala 和普通 dashboard 最不同的地方。

展示内容不应只是数据，还应是“今天真正留下来的东西”：
- 一条高价值 history thread
- 一条 memo
- 一段 long recording summary
- 一个 learned memory

每条 highlight 最好都回答一个问题：
- 你做了什么
- 你决定了什么
- 你记住了什么

### 视觉语气
- `Sala` 要更像一本总结册或工作年鉴的一页。
- 配色建议偏纸感、石墨、浅米、雾蓝，不要 neon。
- 允许使用细线、密度更高的信息编排，让它和 Voko 的“角色舞台”形成反差。
- 强调比例、排版和信息层级，不强调拟人。

### 数据映射建议
第二阶段接真实数据时，优先组合：
- `HistoryStore` 生成今天线程数、时间段、主要主题
- `CollectionStore` 生成 memory highlights
- app name / bundle id 生成 app 占比
- `RunningTaskStore` 生成完成与进行中的闭环情况

### 第一阶段不做
- 严格准确的时长统计
- 复杂跨日趋势
- 真正可解释的心理模型
- 过多图表类型

## 两页之间的关系

| 页面 | 核心问题 | 感受 | 主体 |
|---|---|---|---|
| Voko | “现在你能帮我做什么 / 想起什么？” | 面对一个角色 | 人物 + 任务气泡 |
| Sala | “今天到底发生了什么 / 我是什么状态？” | 翻看一张工作画像 | 总结 + 比例 + memory |

## 视觉对比必须成立

| 维度 | Voko | Sala |
|---|---|---|
| 构图 | 中央聚焦、环绕式 | 编辑排版、块面式 |
| 主角 | 人物核心圆 | 今日画像与图表 |
| 情绪 | 温和、亲近、会回应 | 冷静、总结、会判断 |
| 动效 | 呼吸、漂浮、收音 | 渐显、描边、稳定 |
| 材质 | 哑光陶瓷 + 纸片 bubble | 纸面年鉴 + 细线图表 |
| 阅读方式 | 像在和谁说话 | 像在翻今天的记录册 |

## 实现时的红线
- 不能把 Voko 做成普通 chat layout 加一个圆头像。
- 不能把 Sala 做成一堆同款卡片加图表。
- 不能让两页只靠 icon 和标题区分。
- 不能用一套 accent color 贯穿两页而没有人格差异。
- 不能为了“AI 感”而滥用发光、波形、渐变。

## 第一版建议实现顺序
1. 先把 Voko 做成有舞台感的中心圆 + bubble + mock listening。
2. 再把 Sala 做成 4 个模块：
   hero、app 占比、suggestions、ability hexagon。
3. 最后再把 `History / Collection / Running Task` 接入真实数据。

## 成功标准
- 用户看到 `Voko` 时，会觉得这是 Invoko 里“一个可以找我历史的人”。
- 用户看到 `Sala` 时，会觉得这是“今天的工作画像”，而不是另一个列表页。
- 即使还没接真实语音和真实统计，mock 版本也能把人格和用途传达清楚。
